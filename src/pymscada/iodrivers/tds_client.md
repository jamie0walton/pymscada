# GL-SD-1045 — Market Dispatch Integration (ICCP & Web Services)

## Document Metadata
- **Owner:** Transpower New Zealand Ltd
- **Reference:** GL-SD-1045 / V3
- **Status:** Issued
- **Approval date:** 19-Apr-2023
- **Scope:** Technical specification for Market Dispatch via **Web Services (REST)** and **ICCP**

---

## 1. Channels Supported
- **Web Services (WS)** — RESTful JSON over HTTPS
- **ICCP (TASE.2 / IEC 60870-6)** — Block 2 (status), limited Block 5 (controls)
- Dispatch instructions may be sent over **multiple channels simultaneously**; first valid acknowledgement counts.

---

## 2. Core Concepts

### Dispatch Endpoint
Logical aggregation of Blocks and/or Nodes. All dispatches target an endpoint.

### Dispatch Groups (Product Groups)
- Energy
- Frequency
- Voltage
- Interruptible Load
- DNx (Dispatch Notified)

### Sequence Number
- Maintained per *Dispatch Group + Endpoint*
- Monotonically increasing
- May reset to **1** under defined conditions

### Acknowledgements
- `ACK` – receipt
- `ACKA` – accepted
- `ACKQ` – queried
- `ACKR` – rejected (DNx only)

### 3.2 Interaction Models

#### Server-Sent Events (SSE)
- Long-lived connection
- Server pushes dispatch and heartbeat events

#### Request / Response Polling
- Periodic polling for latest dispatch instructions
- Each poll also acts as a heartbeat

---

### 3.3 Key REST Endpoints (v1)
```
GET  /api/v1/dispatch/endpoints/{endpoint}/instructions
GET  /api/v1/dispatch/endpoints/{endpoint}/instructions/latest
GET  /api/v1/dispatch/endpoints/{endpoint}/instructions/products/{product}/latest
PUT  /api/v1/dispatch/endpoints/{endpoint}/acknowledgements
```

---

### 3.4 Authentication & Security
- **JWT (RS256)** signed with Participant private key
- Mandatory claims:
  - `iss` (issuer)
  - `nbf` (not before)
  - `exp` (expiry ≤ 300 seconds)
- HTTPS only (TLS)
- Transpower retains **public key only**

---

### 3.5 Required HTTP Headers
- `Authorization: Bearer <JWT>`
- `Correlation-Id`
  - Generated by server in dispatch instructions
  - Must be echoed in acknowledgements

---

### 3.6 Dispatch Simulator
- Internet-facing Dispatch Simulator
- Same API and authentication as production
- Uses semi-randomised dispatch data
- Access provided during onboarding

## 4. ICCP Channel

### 4.1 Protocol & Transport
- **ICCP / TASE.2 (IEC 60870-6)**
- TCP/IP transport
- Secure ICCP using:
  - MACE (X.509)
  - TLS with mutual authentication

---

### 4.2 ICCP Blocks
- **Block 1** – Mandatory (association, datasets)
- **Block 2** – Status & dispatch values (required for dispatch)
- **Block 5** – Controls (limited support)

---

### 4.3 ICCP Acknowledgement Mechanism
- Sequence-based data items:
  - `SEQNUM`
  - `ACK_SEQ_NUM`
  - `ACKA_SEQ_NUM`
  - `ACKQ_SEQ_NUM`
  - `ACKR_SEQ_NUM`
- Bad quality treated as `ACKQ`
- Communication failure triggers retry

---

### 4.4 Data Representation
- **Analogs:** IEEE-754 REAL32 (engineering units, no scaling)
- **Timestamps:** Epoch or seconds-since-midnight (per configuration)
- **Quality flags:** Valid / Invalid / Held / Suspect

#### ICCP Naming Rules
- Characters: `A–Z`, `0–9`, `_`
- Upper-case only
- Maximum length: 32 characters
- Names assigned by Transpower

---

## 5. Dispatch Groups & Primary Types

### Energy
- Primary: MW, RESF, RESS, DD
- Secondary: RUP, RDN, IG, GC, FRK, SRK, ND

### Frequency
- Primary: FREQ
- Secondary: FST, FST_C, VALUE_C

### Voltage
- Primary: VOLT, MVAR

### Interruptible Load
- Primary: INTF, INTS

### DNx
- Primary: MWN, DDN

---

## 6. Reliability & Availability
- ICCP service target availability: **≥ 99.9%**

### Minimum Requirements
- Dual independent network paths
- Dual ICCP devices with no single point of failure

### Recommended
- Two geographically diverse sites

---

## 7. Participant Validation Expectations
- Handle sequence rollover (9999 → 1)
- Detect and discard out-of-order dispatches
- Ensure idempotent processing of duplicates
- Discard bad-quality data
- Log missing or duplicate dispatches
